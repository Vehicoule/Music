# Rust Core Migration: Current FastAPI SQLite Contract

This document captures the current SQLite behavior implemented by the FastAPI backend so the Rust core can preserve data compatibility during migration. It focuses on playlists, favorites, history, metadata cache, source metadata indexing, and settings-related database behavior.

## Source files inspected

- `backend/app/core/db.py`
- `backend/app/routes/playlists.py`
- `backend/app/routes/favorites.py`
- `backend/app/routes/history.py` (not present; history routes currently live in `favorites.py`)
- `backend/app/core/config.py`
- `backend/app/main.py`
- `backend/app/services/source_index.py`
- `backend/app/schemas.py`
- `frontend/lib/src/models.dart`

## Startup and connection behavior

- FastAPI constructs a `Database` from `Settings.database_path` during application lifespan startup, then calls `db.init()` before registering stateful services.
- The default database file is `./data/streambox.sqlite3`; it can be overridden through Pydantic settings/environment as `DATABASE_PATH`.
- Every `Database.connect()` call creates the parent directory if needed, opens a new `sqlite3.Connection`, sets `row_factory = sqlite3.Row`, and executes `PRAGMA foreign_keys = ON` for that connection.
- No explicit `PRAGMA journal_mode=WAL`, `busy_timeout`, or connection pooling is configured in the FastAPI code.

## Tables and columns

`db.init()` uses `CREATE TABLE IF NOT EXISTS` for the durable tables below.

### `source_index_fts`

- FastAPI drops this virtual table on every `db.init()` and recreates it as `fts5(source_provider UNINDEXED, source_id UNINDEXED, normalized_text)`.
- It is repopulated from `source_index` immediately after recreation.
- Source-index upserts also delete and reinsert the matching FTS row.
- Treat this table as derived data. Rust may rebuild it rather than migrate it as durable state.

## Primary key formats

- `metadata_cache.cache_key`: caller-defined text keys. Current prefixes include `musicbrainz:v7:recording:...`, ListenBrainz recording popularity keys, yt-dlp source-match keys, YT Music keys, and the source-index version key `source-index:schema-version:v4`.
- `playlists.id`: UUID v4 string from Python `uuid4()`.
- `favorites.id`: UUID v4 string from Python `uuid4()`.
- `history.id`: UUID v4 string from Python `uuid4()`.
- `playlist_tracks`: composite key of playlist UUID string and zero-based integer `position`.
- `source_index`: composite natural key `(source_provider, source_id)`. `SourceIndexEntry.to_track()` exposes these as a track id in the string form `{source_provider}:{source_id}`, but the SQLite key remains two columns.
- `PlaybackItem.id` inside JSON blobs defaults to a UUID v4 string on the FastAPI side. The Flutter client can also create local IDs as `{track.id}-{microsecondsSinceEpoch}` when constructing an item from a selected source.

## Timestamp formats

SQLite timestamp columns are stored as integer Unix epoch seconds in UTC:

- `metadata_cache.created_at`: `int(time.time())` for generic cache writes; `strftime('%s', 'now')` for the source-index schema-version row.
- `playlists.created_at` and `playlists.updated_at`: `int(datetime.timestamp())` from timezone-aware UTC datetimes.
- `favorites.created_at`: `int(datetime.timestamp())` from the `Favorite` model default, which is timezone-aware UTC.
- `history.played_at`: `int(datetime.now(timezone.utc).timestamp())`.
- `source_index.last_matched_at`: `int(time.time())`.

API and JSON blob datetimes are different: Pydantic serializes `datetime` fields inside JSON payloads as ISO-8601 strings. The Flutter client parses `PlaybackItem.added_at` with `DateTime.tryParse()` and serializes it with `toIso8601String()`.

Notes for Rust compatibility:

- `track` is required by both FastAPI/Pydantic and Flutter parsing.
- `source` is nullable/optional.
- `added_at` is generated by FastAPI if omitted when validating a request payload, but rows written by FastAPI use `model_dump_json()` and normally contain it.
- Pydantic may include `null` keys for optional fields when serializing models; Flutter `toJson()` also emits optional keys with null values.
- `SourceCandidate.expires_at` exists in the FastAPI schema and may appear in FastAPI-generated JSON, but the current Flutter `SourceCandidate` model does not parse or re-emit `expires_at`.

### `metadata_cache.payload`

This column stores arbitrary JSON text written with `json.dumps(payload)` and read with `json.loads(payload)`. Known payload families include:

- MusicBrainz search cache: array of serialized `TrackMetadata` objects.
- MusicBrainz artwork cache: object such as `{ "artwork_url": "https://..." }` or `{ "artwork_url": null }`.
- ListenBrainz popularity cache: object with `listen_count` and `listener_count`.
- yt-dlp source-match cache: array of serialized `SourceCandidate` objects.
- YT Music cache: adapter-specific API payloads.
- Source-index schema version marker: key `source-index:schema-version:v4`, payload JSON string `"4"`.

Cache reads return `None` when the key is missing or when `int(time.time()) - created_at > ttl_seconds`; expired rows are not deleted automatically.

## Playlist behavior

- `GET /api/playlists` selects all rows from `playlists` ordered by `updated_at DESC`, then loads each playlist by id.
- Loading a playlist reads `playlist_tracks` ordered by `position` and validates each `item_json` as `PlaybackItem`.
- Missing playlist ids raise HTTP 404 with detail `Playlist not found`.
- `POST /api/playlists` creates a UUID playlist id, sets `created_at` and `updated_at` to the same current UTC time, and persists the provided tracks.
- `PUT /api/playlists/{playlist_id}` loads the existing playlist first. Only non-null fields from the update payload overwrite the current values. `updated_at` is always set to current UTC time before saving.
- Saving a playlist upserts the `playlists` row, preserving the original `created_at` on conflict because only `name`, `description`, and `updated_at` are updated.
- Saving then deletes all existing `playlist_tracks` for that playlist and reinserts the current tracks with zero-based contiguous positions from Python `enumerate()`.
- There is currently no delete-playlist endpoint in `playlists.py`.

## Favorites behavior

- `GET /api/favorites` returns all favorite rows ordered by `created_at DESC` and validates `item_json` as `PlaybackItem`.
- `POST /api/favorites` creates a UUID favorite id and inserts the serialized item. `Favorite.created_at` is generated by the model default at object construction time.
- There is no deduplication by track, source, or item id. Repeated posts can create multiple favorites for the same item.
- `DELETE /api/favorites/{favorite_id}` executes a delete by id and returns 204 even if no row matched.

## History behavior

- There is no separate `backend/app/routes/history.py` file at the time of writing; history endpoints are defined in `backend/app/routes/favorites.py` under the shared `/api` router.
- `GET /api/history` returns up to 100 items ordered by `played_at DESC`; the response model is `list[PlaybackItem]`, so `played_at` and row ids are not returned to clients.
- `POST /api/history` inserts a UUID row id, serializes the provided item, sets `played_at` to current UTC epoch seconds, and returns the item. There is no deduplication or pruning on insert.

## Metadata cache and source-index behavior

- `metadata_cache` is a generic key/value JSON cache with TTL enforced only by readers.
- `set_cache()` upserts by `cache_key`, replacing both `payload` and `created_at`.
- Source-index schema compatibility is tracked through `metadata_cache` key `source-index:schema-version:v4`. During `db.init()`, if that row is missing or its payload is not the JSON string `"4"`, FastAPI deletes all rows from `source_index` and writes the current marker.
- `source_index` has a narrow migration path for older rows: `db.init()` checks `PRAGMA table_info(source_index)` and adds missing columns among `artwork_url`, `source_kind`, `raw_title`, `canonical_title`, `canonical_artist`, and `parse_source`, each as `TEXT NOT NULL DEFAULT ''`.
- The FTS table is intentionally non-durable and is dropped/rebuilt on every FastAPI initialization from current `source_index` rows.
- Source-index search first tries FTS. If FTS search fails with an exception, it returns no FTS results; if no FTS entries are found, it falls back to scanning recent `source_index` rows ordered by `last_matched_at DESC`.

## Settings behavior

- The current backend does not store application settings in SQLite.
- Settings are loaded from defaults, environment, and `.env` using Pydantic `BaseSettings`.
- Database-relevant settings are limited to selecting the SQLite path (`database_path` / `DATABASE_PATH`) and TTLs used when interpreting rows in `metadata_cache`.
- The `/api/debug/runtime` response exposes the resolved database path and selected runtime/tool settings, but those are not persisted into the database.

## Expected behavior when tables do not exist

- Normal FastAPI startup calls `db.init()`, so missing tables are created before routes and services use the database.
- `CREATE TABLE IF NOT EXISTS` does not validate or repair incompatible existing table definitions except for the specific `source_index` column additions listed above.
- If route functions are invoked against a database that has not run `db.init()`, their direct SQL statements will raise SQLite `OperationalError` for missing tables; the route modules do not lazily create tables.
- If `source_index_fts` is missing but `db.init()` has run, it will have been recreated. If it is missing later, source-index upserts or FTS queries can fail, depending on the code path.

## Expected behavior when older data exists

- Existing playlist, favorite, history, and metadata-cache rows remain in place when `db.init()` runs.
- Existing playlist ids, favorite ids, and history ids are not rewritten.
- Existing playlist `created_at` values are preserved by playlist upserts; playlist `updated_at` changes on every update.
- Existing `playlist_tracks` for a playlist are completely replaced whenever that playlist is saved.
- Existing metadata-cache rows remain until overwritten or ignored by TTL. Expired rows are not automatically removed.
- Existing `source_index` rows may be deleted during startup when the schema-version marker is missing or not equal to JSON string `"4"`.
- Older `source_index` tables missing the six known additive columns are altered in place with empty-string defaults before version synchronization.
- Malformed JSON blobs, missing required `PlaybackItem.track`, incompatible timestamp values, or missing required columns outside the narrow `source_index` migration path are not handled by explicit migrations. Reads will fail through JSON decoding, Pydantic validation, Dart parsing, or SQLite errors.

## Can Rust and FastAPI both access the same database during transition?

Yes, but only with the normal SQLite constraints and a shared schema contract:

- Both runtimes may open the same SQLite file if Rust preserves the table names, column types, primary keys, JSON shapes, epoch-second timestamps, and source-index version behavior documented here.
- FastAPI opens short-lived connections per operation and enables foreign keys per connection. Rust should also enable `PRAGMA foreign_keys = ON` on every connection.
- Because FastAPI does not configure WAL or a busy timeout, concurrent access should be treated conservatively: multiple readers are fine, but simultaneous writes can produce SQLite locking contention unless Rust configures compatible busy handling and the deployment coordinates writers.
- During a dual-write or mixed-read/write transition, avoid changing table definitions in a way FastAPI cannot read. In particular, do not remove columns, change primary keys, or change JSON field names while FastAPI is still active.
- Treat `source_index_fts` as derived and owned by whichever runtime initializes/rebuilds it. FastAPI will drop and recreate it on startup, so Rust should not store unique state only in that virtual table.
- A safe transition strategy is: keep FastAPI and Rust on the same schema; let either service read; designate one writer for playlists/favorites/history when possible; and make any Rust migrations backward-compatible until FastAPI is removed.
